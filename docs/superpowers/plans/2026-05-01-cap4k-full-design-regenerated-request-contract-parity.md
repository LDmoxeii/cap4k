# Full Design-Regenerated Request Contract Parity Implementation Plan

> **For agentic workers:** This plan corrects the first implementation attempt. Do not interpret "design-regenerated" as "must be written to `build/generated`". The audit must follow `cap4kPlan` output paths.

**Status:** Implemented and verified in `only-danmuku-zero`.

Completion evidence:

- initial false-positive audit behavior was corrected to use `cap4kPlan` `outputPath` values instead of assuming `build/generated`
- audit accepts both `val` and `var` properties for generated contract fields
- 23 command empty-response drifts were fact-checked by isolating their design entries, regenerating only `designCommand`, and confirming generated output uses `data object Response`
- migrated handlers were updated to keep business logic while matching the generated empty-response contract
- final audit result: 206/206 Query/Cmd/Cli contracts matched, 0 drift
- final compile result: `.\gradlew.bat --no-configuration-cache --no-build-cache compileKotlin` passed
- closure commit in `only-danmuku-zero`: `f1e12f0 fix: align command empty responses with generated contract`

**Goal:** Add an `only-danmuku-zero` dogfood audit that proves every Query/Cmd/Cli `Request` / `Response` contract structure is owned by `codegen/design/design.json` and by the pipeline plan context.

**Architecture:** Keep the audit outside `cap4k` production code. The script reads `codegen/design/design.json`, reads `build/cap4k/plan.json`, maps design entries to `design-command`, `design-query`, and `design-client` plan items, then validates the actual file at each plan item's `outputPath`.

The audit is deliberately output-location agnostic:

- If the plan writes a contract to `src/main/kotlin`, audit that file.
- If the plan writes a contract to `build/generated`, audit that file.
- The audit must not fail merely because a contract is checked in.
- The audit fails when the planned contract cannot be found, contains blocking legacy contract markers, or its `Request` / `Response` structure drifts from plan context.

## Design To Plan Mapping

The audit derives expected contract type names from normalized design input:

| Design tag | Generator id | Type suffix |
| --- | --- | --- |
| `command` | `design-command` | `Cmd` |
| `query` | `design-query` | `Qry` |
| `client` | `design-client` | `Cli` |

For example:

```json
{
  "tag": "query",
  "package": "category",
  "name": "GetCategoryTree"
}
```

Expected plan key:

```text
query|GetCategoryTreeQry
```

The script must then find a plan item whose `generatorId` is `design-query` and whose `context.typeName` is `GetCategoryTreeQry`.

## Audit Rules

Each expected contract is classified with exactly one status:

- `PLANNED_AND_MATCHED`: plan item exists, output file exists, no blocking legacy markers exist, and contract structure matches plan context.
- `MISSING_PLAN_ITEM`: design entry has no matching plan item.
- `MISSING_OUTPUT_FILE`: plan item exists but `outputPath` does not exist.
- `PLAN_CONTEXT_DEFECT`: plan item exists but required context data is missing.
- `LEGACY_CONTRACT_MARKER`: output file contains a blocking old contract marker.
- `CONTRACT_STRUCTURE_DRIFT`: output file exists but package/type/request/response/field structure does not match plan context.

Blocking legacy markers:

- `ListQueryParam<`
- `PageQueryParam<`
- `ListQuery<`
- `PageQuery<`

`Item` is not a blocking marker by itself. New nested response models may legitimately contain an `Item` type.

## Structure Checks

For every matched plan item, the script validates:

- `package <context.packageName>` exists.
- `object <context.typeName>` exists.
- `Request` exists as a class/data class.
- `Response` exists as a data class/data object.
- Every `context.requestFields[].name` appears as a `val` or `var` property.
- Every `context.responseFields[].name` appears as a `val` or `var` property.
- Every request/response nested type name appears as a nested data class.
- If `context.pageRequest == true`, the file contains `PageRequest`, `pageNum`, and `pageSize`.

This is a structural dogfood audit, not a Kotlin parser. It is intentionally lightweight and stable enough for migration review.

## Report Requirements

The Markdown and JSON reports must be deterministic:

- No timestamp.
- Relative paths only.
- Stable sort by `tag`, `packageName`, and `typeName`.
- Include `planFile`, `designFile`, summary counts, status counts, and failure rows.

## Expected Current Result

During migration, the audit may fail. That is acceptable if failures are actionable, for example `CONTRACT_STRUCTURE_DRIFT` or `LEGACY_CONTRACT_MARKER`.

The audit must not produce a blanket failure like "all contracts are missing from build/generated" when the plan actually routes them to checked-in source.
