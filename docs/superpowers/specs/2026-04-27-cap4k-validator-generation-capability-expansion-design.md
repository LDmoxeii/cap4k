# Cap4k Validator Generation Capability Expansion Design

Date: 2026-04-27
Status: Draft for review

## Summary

This slice expands ordinary design validator generation from a minimal fixed-`Long` annotation generator into a structural validator skeleton generator.

The goal is not to generate business validation logic.

The goal is to make validator design input expressive enough to describe:

- annotation targets
- default validation message
- validator value type
- custom annotation parameters

The generated code should compile and provide a stable skeleton that project users can fill in.

This slice works with the analysis design projection normalization spec:

- analysis can project supported validator structure into design JSON
- design JSON can generate the same validator skeleton
- aggregate unique validators remain on the aggregate generator line

## Why This Slice Exists

The current pipeline validator generator was intentionally minimal.

It supports:

- `tag = "validator"`
- package
- name
- description

But canonical assembly hard-codes:

```kotlin
valueType = "Long"
```

The default template also hard-codes:

```kotlin
@Target(AnnotationTarget.FIELD, AnnotationTarget.VALUE_PARAMETER)
class Validator : ConstraintValidator<Annotation, Long>
```

That is enough for a narrow field-level validator, but it cannot represent common project validators such as:

- class-level validators
- validators with `Any` target value
- validators with custom field-name annotation parameters
- validators with custom default messages

The next capability step should make these structural parts explicit while keeping validator body generation deliberately simple.

## Goals

- Expand ordinary `designValidator` input and canonical models.
- Support explicit `targets`.
- Support explicit `message`.
- Support explicit `valueType`.
- Support simple custom annotation parameters.
- Generate compiling annotation + nested validator skeletons.
- Use `return true` as the generated validator body.
- Keep class-level validators reusable through `ConstraintValidator<Annotation, Any>`.
- Keep aggregate unique validators separate from ordinary design validators.

## Non-Goals

This slice will not:

- generate repository/query/client backed validation logic
- generate reflection-based field extraction logic
- automatically attach validators to generated request fields or request classes
- introduce trait/interface-based validator targets
- support concrete generated request classes as validator value types
- support aggregate unique validators through `designValidator`
- merge ordinary validator and aggregate unique-validator models
- support array, enum, or `KClass` custom annotation parameters in the first slice
- support arbitrary Kotlin expressions as parameter default values
- change the Bean Validation runtime contract

## Validator Families

There are two separate validator families.

### Ordinary Design Validators

Ordinary design validators are authored from `designJson` and generated by `designValidator`.

They are application-side validation annotations.

This slice expands only this family.

### Aggregate Unique Validators

Aggregate unique validators are aggregate-derived artifacts.

They are generated from database/aggregate metadata, not from ordinary validator design input.

They must remain separate.

This is important because aggregate unique validators may also be Bean Validation annotations and may also use:

```kotlin
ConstraintValidator<Annotation, Any>
```

That shape alone must not make them ordinary design validators.

## Design Input Contract

An ordinary validator design entry should support this shape:

```json
{
  "tag": "validator",
  "package": "danmuku",
  "name": "DanmukuDeletePermission",
  "desc": "校验弹幕删除权限",
  "message": "无权限删除该弹幕",
  "targets": ["CLASS"],
  "valueType": "Any",
  "parameters": [
    {
      "name": "danmukuIdField",
      "type": "String",
      "defaultValue": "danmukuId"
    },
    {
      "name": "operatorIdField",
      "type": "String",
      "defaultValue": "operatorId"
    }
  ]
}
```

For a field-level validator:

```json
{
  "tag": "validator",
  "package": "category",
  "name": "CategoryMustExist",
  "desc": "分类必须存在",
  "message": "分类不存在",
  "targets": ["FIELD", "VALUE_PARAMETER"],
  "valueType": "Long",
  "parameters": []
}
```

### Required Fields

The canonical validator model should require:

- `package`
- `name`
- `desc`
- `message`
- `targets`
- `valueType`
- `parameters`

Source parsing may provide temporary defaults for older minimal fixtures, but new examples should be explicit.

Recommended defaults:

- `message = "校验未通过"`
- `targets = ["FIELD", "VALUE_PARAMETER"]`
- `valueType = "Any"` when `targets == ["CLASS"]`
- `valueType = "Long"` only as a compatibility bridge for older minimal validator fixtures
- `parameters = []`

The `CLASS -> Any` default reflects the new ordinary class-level validator convention.
Concrete generated request classes are not valid default targets.

## Targets

Supported targets:

- `CLASS`
- `FIELD`
- `VALUE_PARAMETER`

The template maps these to Kotlin:

- `AnnotationTarget.CLASS`
- `AnnotationTarget.FIELD`
- `AnnotationTarget.VALUE_PARAMETER`

Unknown targets should fail fast for user-authored `designJson`.

Reason:

- target spelling is intentional user input
- silently ignoring a target can generate a validator that compiles but is attached to the wrong program elements

The analysis projection path may skip unsupported validators before they become design input.

## Value Type

`valueType` is the second generic parameter of `ConstraintValidator<Annotation, ValueType>`.

Supported first-slice values:

- `Any`
- `String`
- `Long`
- `Int`
- `Boolean`

This list is intentionally small.

It covers:

- reusable class-level validators through `Any`
- common ID validators through `Long`
- text validators through `String`
- enum-code or status validators through `Int`
- boolean flag validators through `Boolean`

Unsupported first-slice values:

- concrete generated request classes
- nested generated request classes
- generated trait/interface targets
- arrays
- enums
- `KClass`
- arbitrary FQCNs that require import resolution

The generator may later support more types, but this slice should not.

## Nullability Rule

`valueType` should not encode Kotlin nullability in the first slice.

For example, design input should use:

```json
"valueType": "Long"
```

not:

```json
"valueType": "Long?"
```

The generated `isValid` signature should always allow null:

```kotlin
override fun isValid(value: Long?, context: ConstraintValidatorContext): Boolean = true
```

This follows Bean Validation convention:

- custom validators usually accept `null`
- null rejection should normally be expressed with `@NotNull`
- the validator's value type describes the non-null value domain, not field requiredness

For `Any`, the generated signature is:

```kotlin
override fun isValid(value: Any?, context: ConstraintValidatorContext): Boolean = true
```

## Message

`message` is a first-class validator field.

It should render into:

```kotlin
val message: String = "..."
```

The user may override it in design input.

If missing, source parsing may default to:

```text
校验未通过
```

`message` should not be modeled as a custom parameter.

Reason:

- Bean Validation requires a `message` attribute by convention
- it is standard annotation metadata
- analysis projection can read it as a well-known field
- custom parameter lists should only contain domain-specific annotation parameters

## Custom Parameters

`parameters` describe custom annotation constructor parameters beyond standard Bean Validation metadata.

The standard Bean Validation fields are always generated and are not custom parameters:

- `message`
- `groups`
- `payload`

Supported first-slice parameter types:

- `String`
- `Int`
- `Long`
- `Boolean`

Each parameter supports:

- `name`
- `type`
- `nullable`
- `defaultValue`

The primary first-slice use case is reusable class-level validators with field-name parameters:

```json
{
  "name": "danmukuIdField",
  "type": "String",
  "defaultValue": "danmukuId"
}
```

which renders as:

```kotlin
val danmukuIdField: String = "danmukuId"
```

### Unsupported Parameter Types

Arrays, enums, and class literals are intentionally deferred.

They are not impossible, but they add several concerns:

- JSON default-value representation
- Kotlin literal rendering
- imports
- `KClass` syntax
- enum FQCN resolution
- analysis projection of default values

The first slice does not need that complexity.

If a project needs those forms, it can override the generated template or keep the validator hand-written until a later parameter-type expansion.

## Generated Code Shape

The generator should continue to emit a single Kotlin file per ordinary validator.

The generated file contains:

- package declaration
- imports
- KDoc from description
- `@Target(...)`
- `@Retention(AnnotationRetention.RUNTIME)`
- `@Constraint(validatedBy = [...])`
- `@MustBeDocumented`
- annotation class
- standard Bean Validation annotation fields
- custom annotation parameters
- nested `Validator`

Example output shape:

```kotlin
package edu.only4.danmuku.application.validators.danmuku

import jakarta.validation.Constraint
import jakarta.validation.ConstraintValidator
import jakarta.validation.ConstraintValidatorContext
import jakarta.validation.Payload
import kotlin.reflect.KClass

/**
 * 校验弹幕删除权限
 */
@Target(AnnotationTarget.CLASS)
@Retention(AnnotationRetention.RUNTIME)
@Constraint(validatedBy = [DanmukuDeletePermission.Validator::class])
@MustBeDocumented
annotation class DanmukuDeletePermission(
    val message: String = "无权限删除该弹幕",
    val groups: Array<KClass<*>> = [],
    val payload: Array<KClass<out Payload>> = [],
    val danmukuIdField: String = "danmukuId",
    val operatorIdField: String = "operatorId",
) {
    class Validator : ConstraintValidator<DanmukuDeletePermission, Any> {
        override fun isValid(value: Any?, context: ConstraintValidatorContext): Boolean = true
    }
}
```

## Validator Body

The first slice should always generate a safe compiling stub:

```kotlin
return true
```

or the expression-bodied equivalent:

```kotlin
override fun isValid(value: T?, context: ConstraintValidatorContext): Boolean = true
```

It should not generate:

- `TODO()`
- reflection field reads
- repository queries
- mediator queries
- external system calls
- generated comments pretending to know business logic

Reason:

- validator logic is frequently project-specific
- it may depend on repositories, query handlers, clients, config, or external systems
- a generic generator cannot infer correct business behavior safely
- `return true` preserves compile success and makes the generated skeleton explicit

Users who need real logic can:

- edit the generated file after generation
- override the validator template
- keep advanced validators as hand-written slot/template files

## Field Requirements And Attachment

Design input may declare custom field-name parameters.

This is allowed:

```json
"parameters": [
  {
    "name": "operatorIdField",
    "type": "String",
    "defaultValue": "operatorId"
  }
]
```

But this slice does not automatically attach validators to request fields or request classes.

It also does not require generated request classes to implement any validator-specific interface.

Reason:

- attachment affects command/query/client/api-payload generators
- generated requests may need multiple validators
- field conflicts and import conflicts need a separate design
- trait-based validation is a broader cross-generator feature

The validator generator only owns the annotation class.

## Canonical Model

The ordinary validator canonical model should become:

```kotlin
data class ValidatorModel(
    val packageName: String,
    val typeName: String,
    val description: String,
    val message: String,
    val targets: List<String>,
    val valueType: String,
    val parameters: List<ValidatorParameterModel> = emptyList(),
)

data class ValidatorParameterModel(
    val name: String,
    val type: String,
    val nullable: Boolean = false,
    val defaultValue: String? = null,
)
```

The existing `valueType = "Long"` hard-code should be removed.

The model should be shared by:

- `designJson` source assembly
- `designValidator` planner/render model
- drawing-board validator output from analysis projection

There should not be a separate legacy validator model path.

## Source Parsing Rules

`designJson` should parse:

- `message`
- `targets`
- `valueType`
- `parameters`

Validation rules:

- unknown targets fail fast
- unsupported `valueType` fails fast
- unsupported parameter types fail fast
- parameter names must be non-blank
- parameter names must not be `message`, `groups`, or `payload`
- duplicate parameter names fail fast

Defaulting rules:

- missing `message` defaults to `校验未通过`
- missing `targets` defaults to `["FIELD", "VALUE_PARAMETER"]`
- missing `valueType` defaults to `Any` when the final target set is only `["CLASS"]`
- missing `valueType` may default to `Long` only as a temporary compatibility bridge for old minimal tests
- missing `parameters` defaults to an empty list

The preferred new style is explicit input.

## Template Rendering Rules

The template should use the existing design template import helper style:

```peb
{{ use("jakarta.validation.Constraint") -}}
{{ use("jakarta.validation.ConstraintValidator") -}}
{{ use("jakarta.validation.ConstraintValidatorContext") -}}
{{ use("jakarta.validation.Payload") -}}
{{ use("kotlin.reflect.KClass") -}}
```

It should render imports through:

```peb
{% for import in imports(imports) %}
import {{ import }}
{% endfor %}
```

The rendered target list should be stable and deterministic.

Recommended order:

1. `CLASS`
2. `FIELD`
3. `VALUE_PARAMETER`

The rendered parameter list should preserve input order after the standard Bean Validation parameters:

1. `message`
2. `groups`
3. `payload`
4. custom parameters in design order

## Request-Type Validator Policy

Concrete request-type validators are not supported.

This is not a missing feature in this slice.
It is an intentional contract decision.

Unsupported:

```kotlin
ConstraintValidator<VideoDeletePermission, DeleteVideoPostCmd.Request>
```

Supported:

```kotlin
ConstraintValidator<VideoDeletePermission, Any>
```

with annotation parameters such as:

```kotlin
val videoIdField: String = "videoId"
val operatorIdField: String = "operatorId"
```

This keeps validators reusable and prevents the validator generator from depending on generated request class layout.

## Future Trait-Based Validation

Trait-based typed validators remain a valid future direction.

Possible future shape:

```kotlin
interface DanmukuDeletePermissionTarget {
    val danmukuId: Long
    val operatorId: Long?
}

class Validator : ConstraintValidator<DanmukuDeletePermission, DanmukuDeletePermissionTarget>
```

Generated request classes could implement the trait.

This would improve type safety over reflection/string field names.

It is not included now because it requires cross-generator coordination:

- validator generator emits trait/interface
- request generators implement traits
- request field names and field types must be validated
- multiple validators may require multiple traits
- trait package and naming rules must be defined
- analysis projection must recover the trait relationship

The first slice intentionally stays with `Any + annotation field-name parameters`.

## Testing Strategy

### Unit Tests

Add or update source parsing tests for:

- explicit `message`
- default `message`
- explicit `targets`
- default field/value-parameter targets
- class-only target defaulting `valueType` to `Any`
- explicit `valueType`
- supported custom parameters
- duplicate parameter names fail
- reserved parameter names fail
- unsupported parameter types fail
- unsupported value types fail
- unknown targets fail

Add or update canonical tests for:

- `ValidatorModel` carries `message`
- `ValidatorModel` carries `targets`
- `ValidatorModel` carries `valueType`
- `ValidatorModel` carries parameters in stable order
- ordinary design validators stay separate from aggregate unique validators

Add or update renderer/template tests for:

- class-level `Any` validator
- field-level `Long` validator
- custom string parameter rendering
- message override rendering
- target ordering
- `isValid(value: T?, ...) = true`
- no extra blank lines around optional parameter blocks

### Functional Tests

Add one compile-level fixture for:

- class-level `Any` validator with field-name parameters
- field-level `Long` validator
- message override

The fixture only needs to prove generated source compiles.
It does not need to prove business validation behavior.

## Recommended Implementation Order

1. Expand API models for validator parameters.
2. Expand `designJson` parsing and validation.
3. Expand canonical validator assembly and remove the fixed `Long` assumption.
4. Expand validator render model.
5. Update the default validator template.
6. Add unit tests for parsing and canonical assembly.
7. Add renderer/template tests.
8. Add compile-level functional coverage.
9. Update README or capability docs after behavior is verified.

This should be implemented after or together with the model parts of analysis design projection normalization.

Reason:

- both slices need the same expanded validator model
- analysis projection should output JSON that `designJson` can already parse

## Risks

### Risk: Generator Pretends To Know Business Logic

Validators often depend on repositories, query handlers, clients, external services, or application config.

Mitigation:

- generate only skeletons
- use `return true`
- keep business logic hand-written or template-overridden

### Risk: Parameter Model Becomes Too Broad

Arrays, enums, and `KClass` parameters are possible, but they multiply parsing and rendering rules.

Mitigation:

- support only scalar parameters first
- fail fast on unsupported types
- defer advanced parameters to a later slice

### Risk: Validator Attachment Expands Scope

Automatically attaching validators to requests requires changes to command/query/client/api-payload generators.

Mitigation:

- this slice does not auto-attach
- field-name parameters are allowed only as annotation metadata
- trait-based validation is deferred

### Risk: Unique Validators Leak Into Ordinary Validator Model

Aggregate unique validators may look structurally similar to ordinary validators.

Mitigation:

- ordinary design validators and aggregate unique validators remain separate families
- analysis projection filters aggregate unique validators by package layout
- this generator does not consume aggregate unique metadata

## Acceptance Criteria

- `designJson` can express ordinary validators with `message`, `targets`, `valueType`, and custom scalar parameters.
- Generated field-level validators compile.
- Generated class-level `Any` validators compile.
- Generated validator bodies return `true`.
- `message` is rendered as the standard Bean Validation message field.
- Custom parameters render after `message`, `groups`, and `payload`.
- `valueType` does not encode nullable spelling.
- Concrete request-type validators are rejected or not accepted as ordinary validator input.
- Aggregate unique validators remain out of ordinary `designValidator`.
- No request generator changes are required for this slice.

## Final Position

This is a bounded structural expansion.

It improves validator generation enough for stable design input and analysis projection without pretending to solve business validation logic.

The correct first target is:

```text
structural validator declaration -> compiling skeleton
```

not:

```text
business validator behavior synthesis
```
