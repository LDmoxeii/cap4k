# cap4k DB Custom Annotation Contract Redesign

Date: 2026-07-20

## Context

cap4k's DB-schema input contract has accumulated overlapping table and column annotations across identity, relation, ORM provider controls, managed fields, enum/type binding, and value-object history. Several annotations are partially implemented or stale:

- `@Count` / `@C` is parsed as `countHint` but is not consumed by relation inference or templates.
- `@Reference` / `@Ref`, `@Relation` / `@Rel`, and `@Lazy` / `@L` mix reference identity, relation generation, and ORM behavior.
- `@AggregateRoot` / `@Root` / `@R` duplicates the rule that a table without `@Parent` is a root.
- `@DynamicInsert` and `@DynamicUpdate` expose ORM-specific controls that cap4k should not own in the DB annotation contract.
- `@Deleted`, `@Version`, marker-only `@Managed`, and `@Inherited` overlap but do not clearly distinguish system-managed fields, declaration placement, and persistence-provider roles.
- `@ValueObject` / `@VO` has been removed from the intended DB table contract, while value objects are handled through value-object manifests.

This spec defines a destructive replacement contract. Compatibility with old DB comments is not required.

## Current Evidence

Current code evidence that motivates the redesign:

- `DbRelationAnnotationParser` parses `@Count` / `@C` into `countHint`.
- `DbSchemaSourceProvider` copies `countHint` into `DbColumnSnapshot`.
- `DbColumnSnapshot.countHint` exists in the API model.
- Focused search shows `countHint` is not consumed by canonical relation inference or default aggregate templates.
- `AggregateRelationInference` currently falls back to a conventional `<parent_table>_id` column when resolving owned parent bindings.
- Current source-db code does not rely on database foreign-key metadata such as imported/exported key metadata.

## Goals

1. Replace the DB annotation contract with a small present-tense set.
2. Separate identity metadata, ownership metadata, managed-field metadata, and ORM/provider behavior.
3. Remove stale aliases and half-implemented annotation surfaces.
4. Make owned parent binding explicit through `@ParentRef`.
5. Avoid physical database foreign-key metadata and constraints.
6. Provide the `@ParentRef`, `@Managed=scope`, and `@Managed=deleted` inputs needed by later owned-cardinality inference.
7. Keep value-object persistence out of this redesign.
8. Keep read-model weak-reference consumption out of this redesign.
9. Fail fast on unsupported old annotations through a generic invalid-annotation path.

## Non Goals

- Do not redesign value-object manifests or value-object storage.
- Do not design read-model relation generation for weak references.
- Do not preserve compatibility for old aliases or old annotations.
- Do not depend on physical DB foreign keys.
- Do not introduce a new relation cardinality annotation such as `@One`, `@Count`, or `@ParentCardinality`.
- Do not redesign soft-delete tombstone SQL. That is covered by `2026-07-20-cap4k-soft-delete-discriminator-policy-design.md`.
- Do not infer owned parent-child cardinality from primary keys or unique constraints. That is covered by `2026-07-20-cap4k-owned-relation-cardinality-from-unique-constraints-design.md`.
- Do not implement code in this spec.

## Annotation Contract

### Table Annotations

Supported table annotations:

| Annotation | Meaning |
|---|---|
| `@Parent=<table>` | Declares the current table as an owned child table of `<table>`. |
| `@Ignore` | Excludes the table from cap4k generation input. |

Rules:

- A table without `@Parent` is an aggregate root by default.
- `@Parent` requires the named parent table to be present in scope unless normal out-of-scope table rules apply.
- `@Parent` does not imply any physical database foreign-key constraint.
- `@Parent` does not identify the parent reference column. The child table must mark that column with `@ParentRef`.
- `@Ignore` excludes the table before canonical aggregate/relation/factory planning.
- No table annotation aliases are supported.

### Column Annotations

Supported column annotations:

| Annotation | Meaning |
|---|---|
| `@ParentRef` | Marks the column that stores the owned parent identity. Valid only on child tables with `@Parent`. |
| `@Type=<TypeName>` | Binds the domain-side field type. |
| `@RefAggregate=<AggregateName>` | Marks a same-context weak reference identity and lets cap4k resolve the referenced aggregate ID type. |
| `@RefId=<TypeName>` | Marks a local/external reference ID type when the target aggregate is not resolved locally. |
| `@IdStrategy=db_identity` | Marks a primary-key column whose value is generated by the database identity mechanism. |
| `@Managed=system` | Marks a system-managed field that is not part of factory/create/update payloads. |
| `@Managed=scope` | Marks a system-managed scope/partition column that may be ignored for owned cardinality unique-constraint inference. |
| `@Managed=deleted` | Marks the soft-delete discriminator field. |
| `@Managed=version` | Marks the optimistic-lock version field. |
| `@Inherited` | Marks a managed field as declared by an entity base class rather than by the generated concrete entity. |

Rules:

- No column annotation aliases are supported.
- `@ParentRef` is marker-only.
- `@ParentRef` is valid only on a table with `@Parent`.
- A child table with `@Parent` must have exactly one `@ParentRef`.
- A non-child table must not have `@ParentRef`.
- `@ParentRef` must not be combined with `@RefAggregate`, `@RefId`, or `@IdStrategy`.
- `@Type` requires a non-blank type name.
- `@RefAggregate` requires a non-blank aggregate name.
- `@RefId` requires a non-blank type name.
- `@RefAggregate` conflicts with `@RefId`.
- `@IdStrategy=db_identity` is valid only on a primary-key column.
- `@IdStrategy` supports only `db_identity` in this contract.
- Bare `@Managed` is invalid.
- `@Managed` supports only `system`, `scope`, `deleted`, and `version`.
- A column may have at most one `@Managed` role.
- `@Inherited` is valid only when the same column has `@Managed=system`, `@Managed=scope`, `@Managed=deleted`, or `@Managed=version`.
- `@Inherited` affects only generated concrete entity scalar declaration. It must not remove the field from schema, canonical metadata, relation inference, projections, provider controls, or diagnostics.

## Identity And Reference Semantics

Identity metadata must stay separate from relation generation.

`@ParentRef` identifies the child table's owned parent identity column. It is an ownership binding marker, not a weak reference, not a physical DB FK declaration, and not an instruction to generate a weak relation.

`@RefAggregate` identifies a weak reference to another aggregate in the same cap4k context. It may be used by type resolution and future read-model metadata, but this spec does not define read-model consumption.

`@RefId` identifies a local or external reference ID type. It does not locate a local aggregate.

`@Type` remains the general field type binding hook. If a column also has `@RefAggregate`, the strong/reference ID type resolution must use the reference contract rather than ad-hoc relation metadata.

`@IdStrategy=db_identity` is a persistence identity strategy for database-generated primary keys. It is not the default aggregate ID path.

## Managed Field Semantics

`@Managed=system` means the field is supplied by system code, database defaults, listeners, interceptors, or template/base-class policy. Factory and application create/update payloads do not collect it.

`@Managed=scope` means the field is a system-supplied scope or partition discriminator. Examples include tenant scope when the tenant is not a business payload field. `scope` fields are excluded from factory/create/update payloads and may be ignored for owned cardinality inference.

`@Managed=deleted` marks the soft-delete discriminator. It replaces `@Deleted` and is excluded from factory/create/update payloads. Soft-delete tombstone SQL is defined by the soft-delete discriminator policy spec.

`@Managed=version` marks the optimistic-lock version field. It replaces `@Version` and is excluded from factory/create/update payloads.

`@Inherited` is a declaration-placement hint for managed fields. If a field is `@Managed=...;@Inherited;`, the default generated concrete entity omits the scalar field/property because a base class or template override is expected to declare it. Other generated artifacts still see the field in canonical metadata.

## Owned Relation Binding

Physical database foreign-key constraints are not part of the source contract. cap4k must not depend on imported/exported key metadata when resolving owned parent-child relations.

Owned binding algorithm:

```text
for each table with @Parent=<parentTable>:
    parentRefColumns = columns marked @ParentRef

    if parentRefColumns is empty:
        fail fast
    if parentRefColumns has more than one column:
        fail fast

    parentRef = parentRefColumns.single
    infer owned relation from parent table to child table using parentRef
```

There is no fallback to `<parent_table>_id`. Explicit `@ParentRef` is required because schemas often use non-conventional parent identity column names.

## Downstream Owned Cardinality Inputs

This iteration defines the DB comment markers that later owned-cardinality inference consumes. It does not perform the inference.

Later relation planning may use:

- the single resolved `@ParentRef` column
- child columns marked `@Managed=scope`
- the child column marked `@Managed=deleted`, if any
- DB primary-key and unique-constraint metadata already available on table snapshots

The full owned-cardinality algorithm, examples, and persistence-shape decisions belong to `2026-07-20-cap4k-owned-relation-cardinality-from-unique-constraints-design.md`.

## Factory And Write Surface

Managed fields remain outside default factory/create/update payloads:

- `@Managed=system`
- `@Managed=scope`
- `@Managed=deleted`
- `@Managed=version`

`@Inherited` does not separately control factory payloads. It only affects generated concrete entity declaration.

For owned children:

- `@ParentRef` is a structural binding field.
- Factory behavior must not rely on delayed persistence failures when the parent reference cannot be constructed.
- The canonical/template context should expose whether a constructor field is structural, managed, inherited, or unresolved so default templates can fail fast or emit an intentional construction stub.

This spec does not decide the final factory payload shape for `ownedCardinality=ONE` child entities. It only requires the canonical model to expose enough metadata for templates to make a deliberate choice.

## Parser And Validation Design

Parser cleanup should use a strict allow-list model:

1. Parse all `@Name` annotations from table and column comments.
2. Validate every parsed annotation name against the supported table or column annotation set.
3. Reject unsupported names with a generic diagnostic.
4. Resolve supported annotation values.
5. Strip only supported annotations from cleaned comments.

The parser should not carry removed annotations as no-op metadata.

The validator and active skill/public docs must use the same supported annotation set. Stale docs in `skills` and `docs/public` should be treated as contract drift.

## Model Changes

Likely API/source model changes:

- Remove `DbColumnSnapshot.countHint`.
- Remove `DbColumnSnapshot.referenceTable`, `explicitRelationType`, and `lazy` when old relation annotations are removed.
- Add `DbColumnSnapshot.parentRef: Boolean`.
- Replace marker booleans `deleted`, `version`, and `managed` with a managed role such as `managedRole: DbManagedRole?`.
- Keep or rename `DbColumnSnapshot.inherited`, but validate it only with managed roles.
- Replace `generatedValueDeclared` / `generatedValueStrategy` with a focused ID strategy field if destructive API cleanup is practical.
- Keep `DbTableSnapshot.uniqueConstraints`; it is required for owned cardinality inference.
- Remove table flags and parser outputs for deleted table annotations.

Likely canonical model changes:

- Keep persistence relation type/shape and owned-cardinality metadata changes out of this iteration.
- Continue carrying inherited/managed fields through canonical metadata even if default concrete entities omit the scalar field.

## Acceptance Criteria

1. Supported DB table annotations are only `@Parent=<table>` and `@Ignore`.
2. Supported DB column annotations are only `@ParentRef`, `@Type`, `@RefAggregate`, `@RefId`, `@IdStrategy=db_identity`, `@Managed=system|scope|deleted|version`, and `@Inherited`.
3. Old aliases and removed annotations fail through a generic unsupported-annotation validation path.
4. Tables without `@Parent` are aggregate roots by default.
5. Tables with `@Parent` fail fast unless exactly one column has `@ParentRef`.
6. Owned parent binding does not use physical DB foreign-key metadata.
7. Owned parent binding does not fall back to `<parent_table>_id`.
8. `@Managed=scope` and `@Managed=deleted` are represented distinctly enough for later relation inference to consume.
9. `@Managed=system` and `@Managed=version` are not described as cardinality-neutral in this spec.
10. `@Inherited` only affects generated concrete entity scalar declaration and does not remove schema/canonical/projection metadata.
11. Factory/template contexts can distinguish structural parent refs, managed fields, inherited fields, and unresolved constructor inputs.
