# cap4k Generated Source Output and Entity Behavior Split Design

> Date: 2026-04-28
> Status: Approved for implementation planning
> Scope: pipeline artifact ownership, built-in generated-source classification, generated Kotlin source output, Gradle source-set integration, aggregate entity behavior-safe template, aggregate behavior scaffold
> Out of scope: user-configurable output-classification DSL, repository backend replacement, aggregate runtime fixes outside the entity template shape, legacy `cap4k-plugin-codegen` changes

## Background

The current pipeline treats almost every generated Kotlin file as checked-in source.

That is useful for bootstrap and user-owned business code, but it is noisy for artifacts that are purely derived from stable inputs. Examples include aggregate entity files after the behavior-safe template change, schema `S*` classes, standard repositories, generated enums, generated converters, and unique-query support classes.

The current exporter model also does not express ownership. A plan item has an output path, but it does not say whether the file is:

- pure generated source that can be overwritten under `build/`
- a checked-in scaffold that should be created once and then left alone
- a user-owned source file that the generator must not replace
- a non-source artifact such as a report or JSON plan

This ambiguity blocks a clean split between generated code and hand-written code.

There is a second related issue in aggregate entities. Moving entity files to generated sources is only safe if user behavior no longer needs to be edited inside the generated entity file. That requires an explicit behavior scaffold contract.

## Goals

Introduce an artifact ownership model that allows pure generated Kotlin sources to live under Gradle build output while user-owned code remains checked in.

The first implementation target is:

- define built-in default artifact classification
- allow selected Kotlin artifacts, including aggregate entity files, to be generated into module-local `build/generated/cap4k/...`
- keep business code and implementation bodies in checked-in source
- generate aggregate behavior scaffolds by default
- implement the aggregate entity-template contract required for entity files to safely move to generated sources

This spec defines the implementation contract. The first implementation uses fixed built-in output classification and output-root resolution. It must not expose project-user output-placement configuration.

## Non-Goals

- Do not add a user-facing artifact output-classification DSL in the first slice.
- Do not make per-table, per-entity, or per-family output classification configurable.
- Do not move handlers with business implementation bodies to `build/`.
- Do not generate controllers.
- Do not solve unit-of-work, repository backend, lazy-loading, or aggregate persistence runtime problems.
- Do not silently delete existing checked-in generated files during migration.
- Do not introduce managed-section semantics for build-generated Kotlin sources.

## Ownership Model

The generator should classify artifacts by ownership, not only by template id or output path.

### Generated Source

Generated source is pure derived code.

It is written under a Gradle build directory, can be overwritten by the generator, should not be committed, and must be registered as a Kotlin source directory for the target module.

Example target shape:

```text
<module>/build/generated/cap4k/main/kotlin/<package-path>/<File>.kt
```

Generated source is appropriate when all meaningful content can be reproduced from stable generator inputs.

### Checked-In Scaffold

Checked-in scaffold is generated once under `src/main/kotlin`.

The generator creates it only when missing. If the file exists, the generator skips it. After creation, the user owns the file.

This is appropriate for files that need a standard starting point but are expected to contain project-specific code.

### User Source

User source is not generated by the pipeline, except through explicit slots or bootstrap inputs selected by the user.

This includes business implementations, custom adapters, controllers, and manually maintained code.

### Output Artifact

Output artifacts are reports or machine-readable intermediate files, not Kotlin source.

Examples include plan reports, analysis projection files, and generated JSON outputs.

## Built-In Default Classification

The first implementation should use built-in defaults. There is no project-level output-classification DSL in this slice.

### Build-Generated by Default

These artifact families are pure enough to be generated under module-local `build/generated/cap4k/...` by default:

- aggregate entity files, after applying the behavior-safe entity template contract in this spec
- aggregate schema classes such as `S*`
- standard aggregate repositories
- generated enum classes from enum manifest, database metadata, or type registry
- enum translation and converter artifacts
- aggregate unique query classes
- aggregate unique query handlers
- aggregate unique validators
- other aggregate-level fully derived query support classes that do not contain user business logic

The important rule is not the file name. The rule is whether the artifact can be fully regenerated without losing user intent. The first implementation applies this rule through framework-owned built-in defaults, not through user configuration.

`Unique*` belongs in this category because it is a standard derived aggregate guard/query family, not a user-authored validator body.

### Checked-In Scaffold by Default

The aggregate behavior scaffold is checked in by default:

```text
<AggregateRootName>Behavior.kt
```

It is generated once, uses a skip/keep-existing policy, and is owned by the user after creation.

The scaffold is enabled by default. Users should not need to opt in to get a place for aggregate behavior.

### Checked-In Source by Default

These artifact families stay checked in by default because they normally contain business or integration code:

- command handlers or command objects that contain implementation bodies
- query handlers that contain hand-written read-model logic
- distributed-client handlers
- ordinary design validators with project-specific validation bodies
- domain event subscribers
- controllers
- API adapters
- mapper implementations
- slot-provided files

If a future generator can prove that a specific handler family is fully derived, that family can be reclassified later. The default must not assume that handlers are safe to overwrite.

### Entity Files

Entity files are generated-source artifacts in this slice.

The move is safe only because this slice also changes the aggregate entity template to the behavior-safe contract below and generates a checked-in behavior scaffold per aggregate root.

The generated entity file must no longer be the natural place for user behavior. It is fully derived aggregate structure and persistence mapping. User behavior belongs in the checked-in `<AggregateRootName>Behavior.kt` file or in user-created checked-in source files.

During migration, the plugin must not delete old checked-in entity files automatically. If a project already has checked-in entity classes with the same FQNs, duplicate classes are an expected migration state until the project owner removes the old files explicitly.

## Behavior Scaffold Contract

Behavior scaffolds are generated per aggregate root, not per child entity.

For an aggregate root `VideoPostProcessing`, the default scaffold is:

```text
VideoPostProcessingBehavior.kt
```

The file lives in the same aggregate package group as the root and can contain extension functions for the root and its owned child entities:

```kotlin
fun VideoPostProcessing.startProcessing() {
    // user code
}

fun VideoPostProcessingFile.markUploaded() {
    // user code
}

fun VideoPostProcessingVariant.finish() {
    // user code
}
```

This avoids generating one empty behavior file for every child entity while still allowing behavior to be split manually later if the aggregate grows.

The scaffold should be minimal:

- correct package declaration
- imports only when necessary
- a short comment explaining that root and owned-entity behavior may be placed here
- no generated business logic
- no managed-section markers

The generator must skip an existing behavior file. It must not merge, rewrite, or format user-owned behavior code.

## Entity Template Contract for Generated Entities

Aggregate entity files become build-generated in this slice, so the template must support a stable behavior split in the same implementation.

The required entity shape is closer to the old plugin output than the current immutable-style generated entity output:

- generate regular `class` declarations, not `data class`
- constructor parameters are initialization inputs, not persisted immutable primary-constructor properties
- persisted scalar fields are declared in the class body and initialized from constructor parameters
- mutable scalar fields use bounded setters such as `internal set`, matching the old-plugin behavior-safe style
- owned collections are behavior-friendly, typically `MutableList<T>` initialized with `mutableListOf()`
- generated relation fields must support aggregate methods mutating child collections safely
- package and visibility must allow the checked-in behavior scaffold to mutate aggregate state within the intended module boundary
- existing aggregate persistence annotations, relation annotations, provider-specific annotations, and id-generator annotations remain generated from canonical metadata
- generated entities must not contain business method bodies or managed sections

Example shape:

```kotlin
class Category(
    name: String,
    parentId: Long? = null,
) : AggregateRoot<Long>() {
    var name: String = name
        internal set

    var parentId: Long? = parentId
        internal set

    val children: MutableList<Category> = mutableListOf()
}
```

The exact annotation set depends on the canonical aggregate model, but the Kotlin class shape must follow this contract. The implementation should use the old plugin output style, such as `only-danmuku/only-danmuku-domain/src/main/kotlin/edu/only4/danmuku/domain/aggregates/category/Category.kt`, as the parity reference for mutability and behavior ergonomics.

## Artifact Plan Model Impact

The pipeline should not implement this as a simple exporter-root switch.

`ArtifactPlanItem` needs to carry an output classification or ownership field. A path alone cannot express whether a file belongs in source, build-generated source, or an output/report directory.

A possible model is:

```kotlin
enum class ArtifactOutputKind {
    CHECKED_IN_SOURCE,
    GENERATED_SOURCE,
    OUTPUT_ARTIFACT,
}
```

The exact name can change, but the concept must exist in the plan.

The plan report should expose:

- artifact id or template id
- module role
- package name
- relative output path
- output kind
- resolved output root
- conflict policy

This makes `cap4kPlan` useful for reviewing generated-source classification and resolved output roots before files are written.

## Export and Gradle Integration

Generated Kotlin source output must be registered with the target subproject's Kotlin source set.

For a module such as `only-danmaku-domain`, generated source should be visible to:

- Gradle `compileKotlin`
- IDE import
- tests that compile against generated classes
- downstream modules that depend on that module

The generated-source root should be deterministic and module-local. A recommended default is:

```text
<module>/build/generated/cap4k/main/kotlin
```

The exporter must resolve each artifact against the artifact's module role and owning Gradle subproject. Root-project build output is not enough because source-set registration is per subproject.

For example:

```text
:demo-domain       -> demo-domain/build/generated/cap4k/main/kotlin
:demo-application  -> demo-application/build/generated/cap4k/main/kotlin
:demo-adapter      -> demo-adapter/build/generated/cap4k/main/kotlin
```

Each subproject that owns at least one `GENERATED_SOURCE` Kotlin artifact must register its own generated source root with the `main` Kotlin source set. A root-project generated directory must not be used as a substitute for per-subproject source-set registration.

The generator also needs to avoid compile-task cycles. Some generation sources can depend on analysis or compiled metadata, while generated source needed by `compileKotlin` cannot depend on that same compile.

That means implementation should distinguish at least:

- pre-compile generated source families
- post-analysis/report/export families

Entity, schema, repository, enum, enum translation/converter, and aggregate unique generated-source families are pre-compile source from the perspective of the owning module. They must be generated before that module's Kotlin compilation consumes them. The exact task names belong in the implementation plan, but this spec requires the boundary to exist.

## Conflict Policy

Generated-source files under `build/` are overwriteable.

Checked-in scaffolds use skip/keep-existing.

The generator must not silently replace checked-in user files because a family was newly classified as generated-source. Migration cleanup is a project decision, not an automatic side effect of running `cap4kGenerate`.

During migration, duplicate classes are possible if a class exists both in checked-in source and generated source. The implementation plan must include an explicit dogfood cleanup step, but the plugin must not delete user files automatically.

## Default Query and Unique Classification

The user-facing rule is:

- pure derived query support may go to build
- business query handling stays checked in

For aggregate unique support, the default generated-source classification includes:

- `Unique*` query/request artifacts
- `Unique*` query handlers when the handler body is derived from aggregate metadata
- `Unique*` validators when the validator is the standard aggregate guard

For ordinary design queries:

- generated request/response contracts stay on the existing checked-in path in this slice unless a later dedicated spec classifies them as pure generated-source contract output
- query handlers stay checked in unless the family is proven fully derived
- hand-written read-model SQL, Jimmer queries, mapper code, or external-service calls stay checked in

Only aggregate unique family handlers whose bodies are completely derived from aggregate metadata belong in generated source. Ordinary design query handlers remain checked-in source unless a later dedicated spec proves a specific family fully derived.

This keeps generated-source classification from swallowing real business logic.

## Default Enum Classification

Generated enums can live in build output by default when their source of truth is stable generator input:

- enum manifest
- database metadata
- type registry

If a project wants to hand-edit enum behavior, that behavior should be placed in checked-in extension/helper files, not inside the generated enum class.

This matches the same ownership rule as entities: generated structure can go to build only when user behavior has a checked-in place to live.

## Verification Requirements

The implementation plan must include verification for:

- generated-source directories are registered for each affected module
- `compileKotlin` sees build-generated classes without requiring checked-in duplicates
- `cap4kPlan` reports output kind and resolved root clearly
- build-generated files are overwritten deterministically
- checked-in behavior scaffold is created once and skipped on the second run
- enum, entity, schema, repository, converter, and aggregate `Unique*` families are classified according to the built-in generated-source defaults
- entity files are moved to module-local build-generated source only together with the behavior-safe entity template contract and compile verification
- no task dependency cycle is introduced between generated-source tasks, analysis tasks, and Kotlin compilation

Dogfood verification should include a cleanup scenario where old checked-in generated files are removed manually and then regenerated into build output.

## Risks

The largest risk is task graph coupling. If a generated-source family depends on analysis output that itself depends on compilation, the plugin can create a compile cycle. The implementation must classify source families by when they can be generated, not only by where they should be written.

The second risk is duplicate classes during migration. This is expected when a project still has old checked-in generated files. The plugin should report paths clearly, but cleanup must remain explicit.

The third risk is incomplete entity-template parity. Entity files move to generated source in this slice, so the behavior-safe mutability contract and checked-in scaffold generation must be verified together before the migration is considered complete.

## Acceptance Criteria

This design is satisfied when:

- artifact ownership is explicit in the plan model
- built-in defaults classify entity, enum, schema, repository, converter, aggregate unique query, aggregate unique query handler, and aggregate unique validator families as generated-source artifacts
- aggregate behavior scaffolds are generated by default as checked-in source
- behavior scaffolds are generated per aggregate root with skip/keep-existing semantics
- entity files are generated under module-local `build/generated/cap4k/main/kotlin` using the behavior-safe old-plugin-style template contract
- each affected Gradle subproject registers its own generated source root with the Kotlin `main` source set
- no user-facing output-classification DSL is required for the first slice
