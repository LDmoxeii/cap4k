# cap4k Aggregate Lifecycle Behavior Contract Design

## Background

`cap4k` already has runtime support for aggregate lifecycle hooks through `DefaultEntityInlinePersistListener`.

Today the listener recognizes only public no-arg member methods declared on the entity class:

- `onCreate()`
- `onUpdate()`
- `onDelete()`
- `onRemove()`

The aggregate generator, however, already creates a separate `*Behavior.kt` file as the handwritten completion surface. That file is generated with `ConflictPolicy.SKIP`, so it is the safer place for project-specific lifecycle behavior than the generated entity file.

The current gap is that Kotlin extension functions in `*Behavior.kt` compile to static methods such as:

```text
ContentBehaviorKt.onCreate(Content)
```

`DefaultEntityInlinePersistListener` does not look for those methods, so projects cannot rely on behavior files for lifecycle hooks even though that is the preferred author surface.

## Problem

The runtime lifecycle contract and the generated behavior completion surface are not connected.

This causes reference projects and user projects to add manual wrapper methods such as:

```kotlin
fun Content.recordDraftCreated() { ... }
```

Those wrappers only exist because `onCreate` cannot currently be implemented as a behavior extension and consumed by the default lifecycle listener.

## Goals

1. Keep existing entity member lifecycle hooks compatible.
2. Let behavior extension lifecycle hooks participate in the existing runtime listener path.
3. Surface lifecycle hook skeletons in generated `*Behavior.kt` files.
4. Avoid requiring users to override `entity.kt.peb` just to use lifecycle hooks.
5. Keep the lifecycle model simple: no new annotation, registry, DSL option, or runtime service.

## Non-Goals

- Do not redesign persist listener dispatch.
- Do not change factory behavior.
- Do not add lifecycle hook configuration to the Gradle DSL.
- Do not make entity template overrides the recommended path.
- Do not update `cap4k-reference-content-studio` in this issue.

## Runtime Design

`DefaultEntityInlinePersistListener` should resolve lifecycle handlers in this order:

### Create

1. entity member method:
   - `Content.onCreate()`
2. behavior extension method:
   - `ContentBehaviorKt.onCreate(Content)`

### Update

1. entity member method:
   - `Content.onUpdate()`
2. behavior extension method:
   - `ContentBehaviorKt.onUpdate(Content)`

### Delete

Deletion has an existing fallback from `onDelete` to `onRemove`. That fallback must remain, but the search order must preserve `onDelete` priority across both member and behavior surfaces.

Correct order:

1. entity member method:
   - `Content.onDelete()`
2. behavior extension method:
   - `ContentBehaviorKt.onDelete(Content)`
3. entity member method:
   - `Content.onRemove()`
4. behavior extension method:
   - `ContentBehaviorKt.onRemove(Content)`

Incorrect order:

```text
Content.onDelete()
Content.onRemove()
ContentBehaviorKt.onDelete(Content)
ContentBehaviorKt.onRemove(Content)
```

That incorrect order would let `onRemove` hide a valid behavior-level `onDelete`.

## Behavior Extension Lookup

For an entity class:

```text
com.acme.demo.domain.aggregates.content.Content
```

the listener should look for:

```text
com.acme.demo.domain.aggregates.content.ContentBehaviorKt
```

and then resolve a public static method with this shape:

```kotlin
fun onCreate(entity: Content)
```

At reflection level, this is:

```text
ContentBehaviorKt.onCreate(Content)
```

The same convention applies to `onUpdate`, `onDelete`, and `onRemove`.

## Cache Design

The existing cache may remain a `Map<String, Method?>`, but cache keys must distinguish:

- entity member lookup
- behavior extension lookup
- method name
- entity class

For example:

```text
member:com.acme.Content.onCreate
behavior:com.acme.ContentBehaviorKt.onCreate(com.acme.Content)
```

The invocation path must distinguish:

- entity member method:
  - `method.invoke(entity)`
- behavior extension static method:
  - `method.invoke(null, entity)`

## Template Design

`aggregate/behavior.kt.peb` should render lifecycle extension skeletons for aggregate roots:

```kotlin
fun Content.onCreate() {
}

fun Content.onUpdate() {
}

fun Content.onDelete() {
}
```

The file already uses `ConflictPolicy.SKIP`, so these skeletons are generated only as the initial handwritten completion surface.

`onRemove` should not be generated by default. It remains supported at runtime for compatibility, but `onDelete` is the primary public lifecycle hook name.

## Testing Strategy

### Runtime Tests

Add or extend `DefaultEntityInlinePersistListenerTest` to verify:

- entity member hooks still work
- behavior extension hooks work when entity member hooks are absent
- entity member hooks take priority over behavior extension hooks
- delete lookup tries behavior `onDelete` before falling back to `onRemove`
- exceptions thrown by behavior extension hooks propagate like member hook exceptions

Behavior extension tests should use a dedicated test entity file and a dedicated behavior extension file so Kotlin really compiles the extension to `TestEntityBehaviorKt`.

### Renderer Tests

Update renderer tests for `aggregate/behavior.kt.peb` to verify:

- behavior scaffold contains `fun Category.onCreate()`
- behavior scaffold contains `fun Category.onUpdate()`
- behavior scaffold contains `fun Category.onDelete()`
- behavior scaffold does not contain generated business logic
- behavior scaffold does not contain `onRemove` by default

### Generator Planner Tests

Existing behavior planner tests should continue to assert:

- behavior artifact is generated only for aggregate roots
- behavior artifact keeps `ConflictPolicy.SKIP`

No planner behavior change is expected unless current tests need assertion updates for the new template output.

## Public Docs

If implementation changes expose new author-facing behavior, update public authoring docs only narrowly:

- mention lifecycle hooks as behavior-file completion points
- avoid turning lifecycle hooks into a broad new feature tutorial

This issue does not require a large documentation rewrite.

## Acceptance Criteria

- `DefaultEntityInlinePersistListener` recognizes entity member hooks and behavior extension hooks.
- Existing entity member hook behavior stays compatible.
- `onDelete` has priority over `onRemove` across both entity and behavior surfaces.
- Generated `*Behavior.kt` scaffold exposes `onCreate`, `onUpdate`, and `onDelete`.
- No entity template override is required for project-specific lifecycle behavior.
- Runtime and renderer tests cover the new contract.
- Public docs are updated only if needed to keep author guidance consistent.
