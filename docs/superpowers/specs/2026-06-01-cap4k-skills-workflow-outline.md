## Reader And Runtime Boundary

The skills bundle is 100% AI-friendly.

It is for agents helping human authors build business systems with cap4k after humans have a reasonable public-doc mental model. The installed bundle must be self-contained at runtime.

The agent must not be required to access:

- cap4k source repository;
- `docs/superpowers/analysis/`;
- `docs/public/`;
- GitHub issues;
- historical specs/plans;
- Context7;
- this design conversation.

The final skills content may be all English if that improves agent execution quality. Human tutorial readability is not a goal.

## Core Philosophy

The skills system is not a linear generic DDD SOP.

It is a cap4k-aware spiral workflow:

```text
business intent
 -> cap4k-aware tactical interpretation
 -> technical design
 -> generator input projection
 -> plan/generation review
 -> handwritten implementation
 -> verification/audit
 -> forced rollback when mismatch appears
```

Later stages must test and revise earlier assumptions. If implementation proves that the model or technical design does not fit cap4k, the agent must stop and return to the right earlier phase.

## Highest-Priority Rule

The skills bundle must repeatedly enforce this rule:

> Skeletons are generated by cap4k. Complex business logic is implemented inside generated skeletons. Do not leave that structure unless the technical design explicitly decides to. If implementation discovers the need to leave it, stop implementation and return to design.

This rule exists because cap4k's generator is a structural drift-control mechanism, not a convenience scaffold. Bypassing it turns cap4k's main value into a maintenance burden.

## Skeleton Generation Gate

Before an agent creates or edits a structural skeleton, it must pass the Skeleton Generation Gate.

Trigger before creating:

- command;
- query;
- client;
- api_payload;
- command handler;
- query handler;
- client handler;
- domain_event;
- integration_event;
- domain event subscriber/handler skeleton;
- inbound integration subscriber skeleton;
- domain_service;
- saga;
- aggregate/entity/relation;
- aggregate projection;
- aggregate factory/specification/unique helper;
- repository/controller/job/projection if current cap4k generator or bootstrap can express it;
- package or directory skeleton;
- a class/interface/handler added just to fix compilation.

The gate must answer:

1. Is this a cap4k generator-supported skeleton?
2. Can the current cap4k version express it through DB/design/types/addon/options?
3. If yes, does the project already have the corresponding generator input?
4. If not, why is the agent not returning to generator input authoring?
5. Has `cap4kPlan` / `plan.json` been reviewed for this skeleton?
6. Will `cap4kGenerate` create or update it?
7. If hand-written, was the exception already recorded in technical design?
8. Does this hand-written path preserve generated-vs-handwritten ownership?

Allowed pass states:

- not generator-supported;
- generated from inputs and reviewed in plan;
- explicitly documented exception from technical design;
- author-owned logic inside generated skeleton;
- explicit user override with risk and verification recorded.

Failure action:

- stop implementation;
- return to technical design or generator inputs;
- update inputs or exception decision;
- review plan again;
- only then continue.

## Forced Rollback Triggers

The final skills design must include hard rollback triggers, not suggestions.

Examples:

- generic DDD concept has no cap4k tactical carrier;
- terms such as strategy, policy, process, validation, orchestration, external callback, or workflow appear without cap4k mapping;
- expected skeleton is missing from `cap4kPlan`;
- generated output path or ownership does not match technical design;
- implementation needs a parallel structure outside generated skeletons;
- agent creates a file to repair compilation instead of returning to generator inputs;
- verification finds layer drift, directory drift, generated/handwritten ownership drift, or logic placed in the wrong layer.

Rollback target examples:

- concept mismatch -> tactical modeling;
- unclear carrier -> technical design;
- missing input -> generator inputs;
- plan mismatch -> generator inputs or technical design;
- implementation bypass -> technical design;
- structure drift -> earliest phase that introduced the wrong assumption.

## Coarse Skill Bundle Shape

The final design may rename everything. Existing skill folder names are not constraints.

Prefer names by authoring phase and execution responsibility, not by current files or framework modules.

Candidate shape:

```text
skills/
  cap4k-authoring-workflow/
  cap4k-business-discovery/
  cap4k-tactical-modeling/
  cap4k-technical-design/
  cap4k-generator-inputs/
  cap4k-generation-review/
  cap4k-handwritten-implementation/
  cap4k-verification-audit/
  shared/
  scripts/
```

The final detailed design must decide whether these are separate skills, workflows under fewer skills, or a hybrid. It must not keep existing names just because they exist.

## Required Self-Contained Knowledge

The installed bundle must include, inside the bundle:

- cap4k tactical affordance map;
- generated skeleton vs handwritten logic policy;
- Skeleton Generation Gate;
- forced rollback triggers;
- business discovery checklist;
- cap4k-aware tactical modeling workflow;
- technical design output contract;
- generator input authoring workflow;
- plan review workflow;
- generation/output ownership review;
- handwritten implementation entry gates;
- verification and audit workflow;
- gotchas that prevent common drift;
- validation script or checklist.

The bundle may include shared rules and references. Single skills do not need to work outside the bundle, but each entry must make bundle-internal dependencies explicit.

## Progressive Loading

Self-contained does not mean every task loads everything.

Rules:

- `SKILL.md` files should route, not teach everything.
- Use `rules/` for stable constraints.
- Use `workflows/` for step-by-step execution.
- Use `references/` for background, affordance maps, gotchas, and source-independent facts.
- Always-read surfaces must be small.
- Task-specific files load only when routed.

## Relationship To Public Docs

Public docs are for humans. Skills are for agents.

Skills may encode assumptions from the public mental model, but must not require the agent to read public docs at runtime. The final skills detailed design should use completed public docs as input, then copy or transform the necessary operational rules into the bundle.

## Relationship To analysis

Analysis is for cap4k maintenance. Skills are for business-project authoring agents.

The final skills detailed design may use completed analysis maps to avoid stale facts, but installed skills cannot depend on analysis files. If a code fact is necessary for skill execution, it must be restated inside the bundle in operational form.

## Validation Direction

The current validation script mostly checks structure and phrases. The new validation should also catch high-risk drift patterns where feasible.

Potential checks:

- no external `analysis`/`public`/issue/Context7 dependency as a required step;
- every workflow that enters implementation references Skeleton Generation Gate;
- forbidden direct handwrite language for generator-supported skeletons;
- route files and skill entries do not duplicate independent routing tables;
- stale facts such as old `validator` design tag, old flow/drawingBoard enabled switches, or wrong design-json variant guidance;
- always-read counts remain controlled;
- bundle self-contained files exist.

The final detailed design must decide which checks are mandatory now and which are deferred.

## Non-Goals

- Do not implement framework features.
- Do not make skills a public tutorial.
- Do not require a single skill folder to work outside the bundle.
- Do not keep existing names or boundaries as constraints.
- Do not require Context7.
- Do not write the detailed skills design before analysis and public rewrites are complete.
- Do not copy historical specs/plans into the skill bundle.
