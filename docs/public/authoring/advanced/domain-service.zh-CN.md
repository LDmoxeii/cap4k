# Domain Service

## 什么时候需要它

- 你遇到的行为明显属于领域，不是 controller 编排、不是 query 组装、也不是外部系统适配。
- 但这段行为又不自然属于某个单独聚合或值对象，硬塞进 `Content` 或 `MediaProcessingTask` 会让聚合职责发胀。
- 这段判断通常围绕多个领域概念的组合约束展开，例如“内容是否在审核通过且媒体已完成后进入可发布判定”，但最后真正改状态的仍然是聚合自己。

在教学项目里，一个典型例子是“发布资格判定规则”。它明显属于领域，不是 application 随便编排；但如果规则既要看 `Content` 的审核状态，也要看 `MediaProcessingTask` 的处理结果摘要，直接把所有判断塞进任何一个聚合都可能变得别扭。

## 为什么默认路径不够

默认路径优先要求行为落在聚合或值对象里，因为这是最容易审计的业务真相面。大多数时候这已经足够，例如送审、审核通过、处理开始、处理成功、处理失败，都应该由 `Content` 或 `MediaProcessingTask` 自己表达。

只有当行为“明显属于领域，但不自然属于某个聚合或值对象”时，领域服务才成立。否则一旦放松，application handler 很容易把任何不想收拾的逻辑都丢进一个所谓 service，最后它沦为 orchestration 垃圾桶。

## 推荐形态

- 先问两个问题：
  - 这段逻辑是不是领域真相，而不是流程调度或外部调用。
  - 它是不是不自然属于单个聚合或值对象。
- 领域服务只负责做领域判断、计算或协同规则表达，不直接接管入口编排。
- 它可以读取 `Content`、`MediaProcessingTask` 或相关值对象，返回一个明确的领域结论，例如“允许发布”“需要等待媒体完成”“需要退回重试”。
- 真正的状态修改仍由聚合行为完成。例如 application handler 先调用领域服务拿到资格结论，再决定是否向 `Content` 发起发布相关命令。
- callback 主路径与 polling 备用路径带回的外部结果，先统一进入 `MediaProcessingTask`；领域服务只看已经内部化后的领域事实，不看入口差异。

## 常见误用

- 把 application orchestration 塞进领域服务，例如在一个 service 里顺手发命令、调 `cli`、写仓储、发消息。
- 因为懒得拆聚合行为，就把几乎所有业务判断都做成 `*DomainService`，让 `Content` 和 `MediaProcessingTask` 只剩数据壳。
- 让领域服务直接依赖 callback payload 或 polling 返回 DTO，而不是依赖已经统一后的领域对象。
- 用“这是领域服务”当借口，把跨聚合写入合法化，绕开 [Default Happy Path](../default-happy-path.zh-CN.md) 的单命令单聚合根边界。

## 审计检查点

- 这段逻辑是否真的明显属于领域，而不是应用层编排或适配层转换。
- 如果把它放回 `Content`、`MediaProcessingTask` 或某个值对象，是否会更自然；如果会，那就不该升成领域服务。
- 领域服务是否只返回领域判断 / 计算结果，而不是自己变成写入主面。
- callback 主路径和 polling 备用路径的差异，是否在进入领域服务前已经被抹平。
- 审阅时能否明确指出：这个 service 解决的是哪一条默认路径表达不下的问题。
