# Read-only Weak Reference

## 什么时候需要它

- 你需要在同一限界上下文里表达跨聚合导航，例如从 `Content` 看到关联的 `MediaProcessingTask` 摘要，或从 `MediaProcessingTask` 看到所属 `Content` 的最小只读信息。
- 这种需求是“看”和“说清关系”，不是“直接拿过去改”。
- 只用裸 ID 已经不足以支撑作者理解和查询导航，但你又不想把另一个聚合硬塞成可写强引用。

在共享教学项目里，这通常出现在内容详情页、处理进度页或人工审阅页上。作者希望在同一个上下文里表达“这篇 `Content` 对应哪一个 `MediaProcessingTask`”，同时保持仓储和写模型边界不被打穿。

具体语境见 [示例总览](../examples/index.md)：只读弱引用讨论仍然围绕 `Content` 与 `MediaProcessingTask` 的关系展开，不另起样例。

## 适用边界

[Default Happy Path](../default-happy-path.md) 默认禁止跨聚合写模型强引用，因为 `Content` 和 `MediaProcessingTask` 仍然是两个不同聚合根。默认做法通常是：写模型里只保留 ID，导航交给查询投影或读模型。

只有当“只靠 ID 不能清楚表达关系，但又不能引入可写强引用”时，只读弱引用才有意义。它解决的是统一类型表达和导航 surface，不是让 repository 突然开始管理跨聚合对象图。

## 推荐形态

- repository 仍然只感知写模型，只负责 `Content` 或 `MediaProcessingTask` 自己的持久化，不负责顺手装配另一个聚合实例。
- 只读弱引用只统一两件事：
  - 类型表达：让人知道这里导航到的是哪类对象。
  - 导航 surface：让作者在领域或读侧表达“有关联，但不可借此写入”。
- 第一阶段推荐“两套类、统一上下文”：
  - 写模型继续保留各自边界清楚的类。
  - 读侧或导航侧另外定义只读引用 / 摘要类。
  - 它们共享同一个业务上下文和命名体系，但不合并成一个可写对象图。
- callback 主路径和 polling 备用路径都只更新自己的写模型真相；跨聚合可见性通过查询或只读导航面补齐，而不是在回写时联动抓取另一个聚合。

## 常见误用

- 看到“弱引用”就让仓储直接返回带完整 `Content` + `MediaProcessingTask` 对象图的写模型。
- 用只读弱引用当借口，在一次命令里顺手修改关联聚合，实质上恢复了跨聚合强引用写入。
- 因为想少写一套读类，就把导航对象和写模型对象混成同一个类，最后 repository 边界和查询边界一起变脏。
- 把 callback 主路径收到的媒体结果顺手写回 `Content` 引用对象，或者让 polling job 在查询导航结构上修状态。
- 没有统一上下文命名，导致一边叫 `ContentMediaRef`，另一边叫 `MediaTaskLink`，审阅时看不出它们表达的是同一种关系。

## 审计检查点

- repository 是否仍只感知写模型，没有因为“弱引用”开始负责跨聚合对象装配。
- 当前设计是否只统一类型表达和导航 surface，而没有把跨聚合写入合法化。
- 是否采用了第一阶段推荐的“两套类、统一上下文”，让写模型和只读导航面保持分离。
- `Content` 与 `MediaProcessingTask` 的生命周期边界是否依然清楚，各自只改各自的真相。
- callback 主路径和 polling 备用路径是否都只推进自己的内部命令，没有在导航结构上偷写状态。
