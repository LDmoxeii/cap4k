# 示例合同

## 何时阅读本页

- 当你已经知道要写哪一层、哪一个概念，但不确定示例必须包含哪些固定部分时，读本页。
- 如果还没有先确认默认规则，请先回到 [Default Happy Path](default-happy-path.zh-CN.md)。

所有作者指南示例都必须遵守同一结构。这个合同的目标不是让示例更像宣传片，而是让项目作者和审阅者都能快速看出“这个例子在讲什么、为什么这样归位、错法长什么样”。

## 五段结构

### Scenario

- 说明这个例子在统一参考项目中的业务场景、前置条件和边界。
- 读者应能立刻知道它属于“内容发布与处理示例项目”的哪一段链路。

### Why this layer / concept

- 说明为什么这个例子属于当前层或当前高级概念。
- 必须把它和 Default Happy Path 的关系讲清楚：是默认写法、默认例外，还是高级偏离。

### Recommended shape

- 给出推荐目录、对象、命令链、事件链、边界转换或结构轮廓。
- 这部分至少要让读者知道应该把代码写到哪里，而不是只给一句抽象原则。

### Non-example / misuse

- 必须给出错误归位、越层写法、错误建模或错误编排。
- 错误示例不是可选补充，而是帮助审阅者快速定位偏离的必要部分。

### Audit cues

- 给出最短审阅路径：看哪些对象、看哪条命令链、看哪个目录，就能判断是否还在规则内。
- 审计线索必须能被人类 reviewer 直接执行，而不是变成空泛口号。

## 固定模板

```markdown
## Scenario
## Why this layer / concept
## Recommended shape
## Non-example / misuse
## Audit cues
```

## 高级概念示例的窄扩展

- 上面的五段结构仍然是所有示例页的基线合同，不因为高级概念示例而改写。
- 如果是高级概念示例页，可以在 `Non-example / misuse` 与 `Audit cues` 之间补一个 `Usage boundary` 小节，用来说明“什么时候该用、什么时候还不该升级”。
- 高级概念示例页在确实有助于澄清边界时，也可以补充短小的支持性片段；但片段只负责辅助说明，不能替代 `Scenario`、`Why this layer / concept`、`Recommended shape`、`Non-example / misuse`、`Audit cues` 这些叙述性合同。
- 其他示例页仍按原五段结构执行，不因为这条窄扩展而默认增加额外章节。

## 示例不是随手代码片段

- 必须说明它在统一参考项目中的位置。
- 必须说明为什么归到这一层或这一高级概念。
- 必须给出错误写法，而不是只给“正确答案”。
- 必须保留足够的目录、命令、事件或边界信息，让读者知道如何把例子接回真实项目。
