# Tactical Affordance Map

| Business Signal | Cap4k Carrier | Use When | Do Not Use When | Generator Input Surface | Expected Plan Evidence | Handwritten Logic Location | Verification Evidence | Rollback Trigger |
|---|---|---|---|---|---|---|---|---|
| state-changing intent | Command | A user, policy, event reaction, or scheduled trigger asks the system to change business facts. | The use case only reads data or formats a view with no state transition. | `design/design.json` `command` block or aggregate-derived command input. | `generatorId` and `templateId` match the command family, `outputKind` places the skeleton in the application module, and `conflictPolicy` protects handwritten slots. | Command handler or generated command slot in the application layer; aggregate behavior keeps domain invariants. | Static review shows command-query split, handler delegates persistence through Unit of Work, and focused evidence covers state transition and invariant checks. | Query mutates state, handler owns domain invariant, or expected command skeleton is missing from plan evidence. |
| read-only view | Query | The request observes current state, builds a read model, or shapes data for display. | The path changes aggregate state, triggers irreversible side effects, or advances workflow state. | `design/design.json` `query` block, query artifact selection, or read-model projection input. | Plan item uses query or query-handler family, expected module/package placement, and read-only output kind. | Query handler, query adapter, or projection reader in the application/adapter boundary without aggregate mutation. | Static review shows no Unit of Work persistence intent and focused evidence covers response shape and filters. | Query handler persists, emits business events, or carries command semantics. |
| business lifecycle owner | Aggregate | A cluster owns identity, consistency rules, lifecycle state, and behavior that must change together. | Data is only a read shape, external payload, pure value, or cross-aggregate workflow coordination. | DB/schema aggregate markers, aggregate generator input, and optional design alignment for commands/events. | Plan evidence includes aggregate, entity, repository, factory, and projection family items in domain-oriented output roots. | Aggregate methods, factory rules, and domain tests; application handlers only orchestrate loading and saving. | Static review shows invariants stay in domain and tests cover lifecycle transitions without adapter dependence. | Application or adapter becomes lifecycle owner, or schema/generator input marks the wrong aggregate root. |
| identity/value semantics | Value Object or Strong ID | A concept is defined by value equality, stable ID semantics, type safety, or JSON-backed structured value. | The concept owns lifecycle behavior, independent persistence identity, or cross-step process state. | Value-object manifest, enum manifest, DB type markers, or strong ID conventions. | Plan evidence includes type/value-object/enum output with correct package and ownership classification. | Value type methods, validation, parsing, and domain usage sites; no protocol or persistence logic inside the value type. | Static review shows value equality and type-safe usage; focused evidence covers invalid value rejection and serialization mapping if relevant. | Value type grows lifecycle responsibilities or is bypassed by primitive strings/IDs across core logic. |
| internal fact | Domain Event | A domain fact happened inside the bounded context and internal reactions may follow. | The fact is a stable external contract or inbound external message. | DB/schema or `design/design.json` `domain_event` block tied to aggregate/domain context. | Plan item uses domain-event and subscriber-related family where expected, with package derived from domain/application placement. | Domain behavior raises facts; domain/application subscribers react by delegating to commands or policies. | Static review distinguishes internal fact from published language; evidence shows event release aligns with Unit of Work lifecycle. | Domain Event is exposed as an external contract or subscriber becomes the central business decision owner. |
| external published fact | Integration Event | A fact crosses service, team, bounded context, callback, or messaging boundary and needs stable published language. | The fact is only internal domain notification or direct method call inside one bounded context. | `design/design.json` `integration_event` block with inbound/outbound variant and explicit subscriber selection when needed. | Plan item uses integration-event family; inbound subscribers appear only with inbound event selection; fields avoid `resultFields`. | Application inbound subscriber handles typed fact interpretation, idempotency, translation, and command delegation; outbound mapping lives in application reaction. | Static review confirms transport/application split, inbound/outbound distinction, and published-language field stability. | External transport is assigned to business subscriber, internal entity shape leaks outside, or inbound subscriber lacks command delegation boundary. |
| reaction after event | Subscriber | An internal or typed external event should trigger thin application reaction, filtering, command delegation, or follow-up request. | The reaction owns long-running state, retry/compensation workflow, or core aggregate invariant. | `design/design.json` domain/integration event block plus explicit subscriber artifact selection where required. | Plan evidence includes domain-subscriber or integration-subscriber family with event linkage and application placement. | Subscriber class in application layer; complex state change returns to command and aggregate behavior. | Static review shows subscriber is thin, idempotent where external, and delegates writes through command/application path. | Subscriber becomes workflow owner, mutates aggregate directly, or handles protocol parsing. |
| long-running/recoverable process | Saga | Coordination needs persisted progress, retry, recovery, compensation, or explicit multi-step process state. | The flow is a simple synchronous reaction, callback-first state update, or stateless command chain. | `design/design.json` `saga` block and related task/process design inputs. | Plan item uses saga family with application placement and expected task/process skeletons. | Saga class and task/process handlers in application layer, delegating domain changes to commands and aggregates. | Static review shows process state, compensation/retry semantics, and focused evidence covers recovery or compensation paths. | Saga is used only because there are multiple steps, or callback semantics are invented without runtime evidence. |
| scheduled or polling trigger | Scheduled Reaction | Time, cron, timeout, compensation scan, or polling fallback should initiate application behavior. | The trigger is a user command, external callback, or domain event reaction with no time-based boundary. | Technical design exception, addon/options, job/scheduled skeleton input, or explicit generated job support. | Plan evidence shows job/scheduled output where supported, or technical design records a handwritten exception. | Application job/reaction class that reads external state if needed and converges to command semantics. | Static review shows polling is fallback or scheduled policy, not a second source of business truth. | Scheduled job writes aggregate directly, duplicates callback truth, or exists only to patch missing event modeling. |
| cross-service call | External Capability | Application logic needs to request an external system through an anti-corruption boundary. | The dependency is an internal domain service, repository read, or protocol controller entrypoint. | `design/design.json` `client` block, client/client-handler artifact selection, or adapter capability input. | Plan evidence includes client and client-handler family with adapter/application boundary placement. | Application-facing request model and adapter/client handler translate protocol details outside the domain. | Static review shows external protocol fields do not leak into aggregate or value objects unless translated. | Aggregate calls external service directly or adapter decides business truth. |
| stable external contract | Open Host Service / Published Language | Other systems depend on stable commands, callbacks, payloads, events, or vocabulary at the boundary. | The shape is internal DTO churn, private aggregate state, or one-off adapter convenience. | API payload blocks, integration event blocks, controller/client inputs, and documented published-language design. | Plan evidence includes api-payload, controller/client, or integration-event family with expected output roots and conflict policy. | Adapter maps protocol shape; application/domain own semantics and compatibility decisions. | Static review confirms boundary terms are stable, version-aware, and not internal aggregate dumps. | Contract exposes internal fields, changes without compatibility review, or conflates adapter payload with domain model. |
| pre-save constraint | Specification | A domain rule must be checked before persistence or uniqueness/eligibility must guard saved aggregate state. | The rule is a one-off UI validation, read filter, or post-save reaction. | DB/schema uniqueness markers, aggregate/specification generator input, or explicit technical design exception. | Plan evidence includes specification or unique helper family, or a documented exception for handwritten rule placement. | Specification class/domain rule slot; Unit of Work interception enforces pre-save behavior where supported. | Static review shows rule is checked before save and focused evidence covers conflict/invalid cases. | Constraint is only checked in controller, query path, or after persistence. |
| domain collaboration not owned by one aggregate | Domain Service | A domain decision spans aggregates or policies but still belongs to domain language and has no single aggregate owner. | The logic is application orchestration, external protocol translation, persistence query, or simple aggregate method. | `design/design.json` `domain_service` block or technical design domain-service exception. | Plan item uses domain-service family with domain module placement and protected handwritten slot. | Domain service implementation, called by application orchestration while aggregates keep their own invariants. | Static review shows domain vocabulary, no adapter/runtime dependency, and tests cover collaboration rule. | Domain service becomes transaction script, repository facade, or external client wrapper. |
